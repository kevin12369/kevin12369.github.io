## 第一部分：如何把Token消耗砍到90%

### 问题：为什么Token消耗这么快？

先说个真事儿。我刚开始用OpenClaw时，有个对话是这样的：

```
我：你好
龙虾：你好！我是你的AI助手，有什么我可以帮你的吗？
我：帮我写个Python函数
龙虾：好的，我来帮你写一个Python函数。请问你需要什么功能的函数？
我：计算斐波那契数列
龙虾：好的，斐波那契数列是一个经典的数学问题...（后面是一大段解释）
我：直接写代码吧
龙虾：好的，以下是Python代码...（代码+详细注释）
我：谢谢
龙虾：不客气！如果你还有其他问题，随时问我...
```

<Label>问题在哪？</Label>

每次对话，OpenClaw都会：
- 记住整个对话历史
- 重复解释上下文
- 生成冗长的回复

结果就是：Token消耗爆炸。

### 解决方案1：斜杠命令快速"瘦身"

OpenClaw提供了一些内置命令，可以快速清理对话上下文：

```bash
/clear          # 清除当前会话的所有上下文
/snapshot       # 保存当前状态到快照
/restore [ID]   # 恢复到某个快照
/status         # 查看当前Token使用情况
```

<Card title="我的使用习惯">
我每天开始工作前，会先输入 `/clear` 清理昨天的对话上下文。

如果某个对话很重要，我会用 `/snapshot` 保存状态，用的时候再用 `/restore` 恢复。

这样既能保持连续性，又不会让上下文无限增长。
</Card>

### 解决方案2：用memory-search替代"无限长对话"

OpenClaw有一个很强大的功能——`memory-search`，它能让AI从你的知识库中检索信息，而不是记住所有对话。

<Aside>
**memory-search的优势**：
- 不需要记住整个对话历史
- 只检索相关信息
- 大幅降低Token消耗
- 响应速度更快
</Aside>

配置方法很简单：

```yaml
# config.yml
memory:
  type: "vector"
  embedding_model: "text-embedding-3-small"
  vector_store: "sqlite"
  chunk_size: 1000
  chunk_overlap: 200
```

### 解决方案3：多Agent分工，从架构层面降低消耗

如果你的任务比较复杂，可以考虑用多个Agent分工，而不是让一个Agent做所有事情。

<Tabs>
  <TabItem label="低效方式">
**一个Agent做所有事**：

```
用户：帮我整理邮件，然后写报告，最后发给我

Agent1：我需要处理邮件、写报告、发送邮件
      （上下文包含所有任务，Token消耗大）
```
  </TabItem>
  <TabItem label="高效方式">
**多Agent分工**：

```
用户：帮我整理邮件，然后写报告，最后发给我

Agent1（邮件）：我负责整理邮件
Agent2（写作）：我负责写报告
Agent3（发送）：我负责发送邮件
      （每个Agent只处理自己的任务，上下文小）
```
  </TabItem>
</Tabs>

### 解决方案4：按需选择模型

不同任务用不同模型，能省不少钱。

<Card title="模型选择指南">

**简单任务**（如：翻译、摘要）
- 使用 `gpt-3.5-turbo` 或 `claude-3-haiku`
- 成本：$0.0005/1K tokens

**中等任务**（如：代码生成、数据分析）
- 使用 `gpt-4` 或 `claude-3-sonnet`
- 成本：$0.03/1K tokens

**复杂任务**（如：复杂推理、创意写作）
- 使用 `gpt-4-turbo` 或 `claude-3-opus`
- 成本：$0.01/1K tokens
</Card>

### 解决方案5：启用提示缓存

OpenClaw支持提示缓存（prompt caching），对于重复的查询，可以大大降低Token消耗。

```yaml
# config.yml
cache:
  enabled: true
  type: "redis"
  ttl: 3600  # 缓存1小时
  max_size: 1000
```

<Card>
**实际效果**

开启缓存后，我的重复查询成本降低了**70%**。

比如：我每天都会问"今天的天气怎么样"，第一次查询需要消耗tokens，后续查询直接从缓存返回，几乎不消耗tokens。
</Card>

### 成本控制：设置预算上限

为了防止意外超支，我设置了月度预算上限：

```yaml
# config.yml
budget:
  enabled: true
  monthly_limit: 50  # 每月50元
  warning_threshold: 0.8  # 达到80%时警告
  block_threshold: 0.95  # 达到95%时阻止
```

<Collapse title="我的Token消费监控">
我每天都会用 `/status` 查看Token使用情况：

```
Token使用情况：
- 今日消耗：125,000 tokens ($0.75)
- 本月累计：2,500,000 tokens ($15.00)
- 预算使用：30%（$15/$50）
```

这样我能清楚地知道自己的消费情况，不会意外超支。
</Collapse>

## 第二部分：如何让"龙虾"固若金汤

### 问题：安全隐患无处不在

OpenClaw的安全问题，真的让人头大。

我见过太多这样的案例：

<Card title="真实安全案例">
**案例1：公共端口暴露**
某用户为了方便，让OpenClaw监听0.0.0.0:8080（所有网络接口）。结果，任何知道他IP地址的人都能访问他的AI助手。

**案例2：无身份验证**
默认情况下，OpenClaw的管理界面没有密码保护。攻击者可以直接访问并修改配置。

**案例3：Skills投毒**
有人从第三方下载了一个"自动备份"技能，结果这个技能偷偷把他的文件上传到了攻击者的服务器。

**案例4：权限过大**
OpenClaw被授予了root权限，结果误操作删除了系统文件，整个服务器瘫痪。
</Card>

### 解决方案1：端口防护三件套

**1. 不要暴露公共端口**

```yaml
# config.yml
server:
  host: "127.0.0.1"  # 只监听本地
  port: 8080
  # 如果需要远程访问，使用反向代理 + SSL
```

<Aside>
**如果需要远程访问**，建议使用：
- VPN（虚拟专用网络）
- SSH隧道
- 反向代理（如nginx）+ SSL证书
</Aside>

**2. 修改默认端口**

```yaml
# config.yml
server:
  port: 18456  # 使用随机端口，增加攻击难度
```

**3. 启用身份验证**

```yaml
# config.yml
auth:
  enabled: true
  type: "bearer"
  secret: "your-secret-key-here"
  algorithms: ["HS256"]
```

### 解决方案2：Skills风险管理

ClawdHub上的技能成千上万，但不是所有技能都安全。

<Label>Skills安全检查清单</Label>

- ✅ 只从官方ClawdHub下载
- ✅ 查看技能的Star数和下载量
- ✅ 查看技能的代码仓库
- ✅ 检查技能的权限需求
- ❌ 不要安装来源不明的技能
- ❌ 不要安装需要过多权限的技能

<Card title="我的Skills安全实践">
我只安装了10个技能，每个都是：
1. 官方ClawdHub上Star数超过1000
2. 有活跃的代码仓库
3. 最近一个月有更新
4. 只请求必要的权限

对于每个新技能，我都会先在沙箱环境测试一周，确认安全后再在生产环境使用。
</Card>

### 解决方案3：建立高危命令黑名单

有些命令太危险，不应该让AI执行。

```yaml
# config.yml
commands:
  blacklist:
    - "rm -rf /"
    - "rm -rf /*"
    - ":(){:|:&};:"  # fork bomb
    "dd if=/dev/zero of=/dev/sda"
    "chmod -R 777 /"
  require_confirmation:
    - "sudo"
    - "mv"
    - "rm"
    - "wget"
    - "curl"
```

<Aside>
**二次确认机制**

对于高危命令，我设置了二次确认：

```
我：删除所有日志文件

AI：检测到高危命令：rm -rf /var/log/*
   这个操作不可逆，请确认：[y/N]
```

这样能有效防止误操作。
</Aside>

### 解决方案4：不要授予管理员/root权限

这是最重要的安全原则。

```bash
# 不要这样运行
sudo ./openclaw

# 应该这样运行
./openclaw  # 用普通用户运行
```

如果OpenClaw需要某些权限，只授予必要的权限：

```bash
# 只授予必要的目录权限
chown -R user:group /path/to/openclaw
chmod 750 /path/to/openclaw
```

### 解决方案5：在沙箱/虚拟机中运行

即使配置再好，也有出问题的时候。

<Card title="我的部署方案">
我用Docker运行OpenClaw：

```yaml
version: '3'
services:
  openclaw:
    image: openclaw/openclaw:latest
    container_name: my-openclaw
    ports:
      - "127.0.0.1:8080:8080"  # 只监听本地
    volumes:
      - ./data:/app/data
      - ./skills:/app/skills
    environment:
      - CLAW_API_KEY=your-api-key
    restart: unless-stopped
```

这样即使出问题，也不会影响我的主系统。
</Card>

### "六要六不要"安全建议

基于我的实战经验，总结出"六要六不要"：

<Card title="六要六不要">

**✅ 六要：**
1. **要**加强权限管理，只允许访问必要目录
2. **要**优先通过加密通道接入（VPN/SSH）
3. **要**建立日志监控机制
4. **要**定期更新OpenClaw到最新版本
5. **要**定期备份重要数据
6. **要**在沙箱/虚拟机中运行

**❌ 六不要：**
1. **不要**授予管理员/root权限
2. **不要**随意映射端口到公网
3. **不要**安装未经审核的第三方技能
4. **不要**使用第三方修改版或破解版
5. **不要**存储敏感信息（密码、API密钥）在OpenClaw中
6. **不要**在生产环境直接部署，先在测试环境验证
</Card>

## 第三部分：我的踩坑经历

### 踩坑1：一开始没设置预算上限

刚开始用OpenClaw时，我没有设置预算上限，结果某天调试代码，不知不觉消耗了200元Token。

<Label>教训</Label>

**一定要设置预算上限！**

我现在的配置是每月50元上限，达到80%时警告，达到95%时阻止。

### 踩坑2：信任了第三方技能

有一次，我从GitHub上下载了一个"自动部署"技能，结果它偷偷给我的服务器装了个挖矿程序。

<Label>教训</Label>

**只从官方ClawdHub下载技能，而且要检查代码！**

现在我每个技能都会：
1. 查看代码仓库
2. 检查代码质量
3. 在沙箱测试一周

### 踩坑3：没有定期备份

某天OpenClaw突然崩溃，我的所有配置和记忆都丢失了。

<Label>教训</Label>

**一定要定期备份！**

我现在每天凌晨2点自动备份：
```bash
# crontab -e
0 2 * * * /usr/bin/rsync -av /path/to/openclaw /backup/
```

### 踩坑4：忽视了日志监控

有一次，我的OpenClaw被攻击了，但我根本不知道，直到一周后才从日志中发现异常。

<Label>教训</Label>

**一定要监控日志！**

我现在每天都会查看OpenClaw的日志，重点关注：
- 异常的API调用
- 异常的文件访问
- 异常的网络连接
